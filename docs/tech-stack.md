# 技術方針

## 現時点の推奨

まずはWebアプリとして作る。

理由は、開発を始めやすく、スマホ・PCの両方で確認しやすく、後からPWAやスマホアプリ化を検討しやすいため。

最初からモバイル対応を意識し、レスポンシブUIとPWA化しやすい構成にする。

## 推奨スタック

### フロントエンド

- React
- TypeScript
- Vite

### スタイリング

- 最初はCSS Modulesまたは通常のCSS
- モバイルファーストでレイアウトを作る
- タップしやすいボタンサイズと余白を確保する
- UIが増えてきたらTailwind CSSやコンポーネントライブラリを検討

### モバイル対応

Webアプリとして始めても、モバイル対応は可能。

最初から以下を守る。

- スマホ幅を基準に画面設計する
- PCでは横幅を広げすぎず、学習に集中できる幅に制限する
- タップ操作を前提にする
- 学習カードやクイズのレイアウトを画面幅に応じて変える
- 後からPWA化できるように、画面遷移とデータ保存を整理しておく

### データ管理

MVPでは、教材データをローカルのJSONまたはTypeScriptデータとして持つ。

例:

- 文字データ
- 単語データ
- 例文データ
- クイズデータ

### 進捗保存

MVPではブラウザのlocalStorageを使う。

学習データが増えたり、ログイン機能が必要になったら、IndexedDB、Supabase、Firebaseなどを検討する。

### 音声

MVPでは音声なしで作る。

ただし将来的に音声を追加できるよう、教材データには音声ファイルを紐づけられる余地を残す。

TTSの利用も検討できるが、タイ語の自然さや声調の正確さが重要なため、最終的には収録音声または品質の高い音声生成を使いたい。

## 将来の選択肢

### PWA

Webアプリをスマホのホーム画面に追加できるようにする。

MVPの後に、オフライン学習、アプリアイコン、スプラッシュ画面、キャッシュ戦略を検討する。

### React Native

スマホアプリとしてApp StoreやGoogle Playに出す場合の候補。

Web版で学習体験を固めた後、必要であればReact NativeまたはCapacitorを検討する。

### バックエンド

ユーザー登録、進捗同期、有料プラン、AI会話練習が必要になった時点で検討する。

候補:

- Supabase
- Firebase
- 独自API

## 避けたいこと

- 最初から機能を詰め込みすぎる
- 教材データと画面ロジックを強く結合する
- 単語データや将来追加する音声データの差し替えが難しい構造にする
- ログインや課金をMVPの必須条件にする
